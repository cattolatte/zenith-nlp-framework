# ADR-0006: Serving is a thin layer over the Generator; streaming is SSE

## Status
Accepted.

## Context
Phase 5 exposes a trained model over HTTP with streaming. Choices: transport for
streaming, where generation logic lives, and how a byte-level tokenizer streams
without emitting broken characters.

## Decision
1. **Serving is a thin adapter over `Generator`.** The web layer parses a request,
   calls `Generator.generate` / `Generator.stream`, and formats the response — no
   decoding logic in `serving/`. The checkpoint is loaded once and cached.
2. **Streaming uses Server-Sent Events (SSE).** SSE over FastAPI's
   `StreamingResponse` needs no extra dependency and is widely supported; each
   event is a JSON `{"text": chunk}`, terminated by a `[DONE]` sentinel.
3. **Chunks are UTF-8-complete.** `Generator.stream` buffers generated bytes and
   only yields when they decode cleanly, so a partial multibyte character is never
   sent (important for the byte-level tokenizer). Any trailing bytes are flushed
   with replacement at the end.
4. **FastAPI/uvicorn are optional extras**, imported lazily, so importing
   `zenith.serving` — and `zenith` itself — needs no serving dependencies.

## Consequences
- The same decoding controls (temperature/top-k/top-p/penalty/beams) are available
  over HTTP and in the library, with a single implementation.
- Streaming is transport-simple and dependency-light; a richer protocol can be
  added later without changing the generator.
